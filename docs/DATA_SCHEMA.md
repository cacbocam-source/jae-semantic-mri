# Data Schema

## Semantic MRI of Agricultural Education Pipeline

Author: Christopher Clemons
Schema Version: 1.0

---

# 1. Overview

This document defines the canonical data schema used by the Semantic MRI pipeline.

The schema specifies:

* corpus storage structure
* metadata manifest format
* processing outputs
* vector embedding storage
* analysis outputs

This schema ensures that each pipeline phase produces deterministic and compatible outputs.

---

# 2. Data Lake Structure

The project data lake is located at:

```
data/
```

Directory layout:

```
data
│
├── raw
│   ├── Route_A_Modern
│   └── Route_B_Legacy
│
├── processed
│
└── manifests
```

---

# 3. Raw Corpus Storage

Raw documents are stored in:

```
data/raw/
```

Subdirectories represent corpus acquisition routes.

```
Route_A_Modern
Route_B_Legacy
```

### Route A — Modern Sources

Expected formats:

* PDF
* XML
* TXT

Typical sources:

* modern journal repositories
* digital archives
* publisher datasets

---

### Route B — Legacy Sources

Expected formats:

* scanned PDF
* OCR-derived text
* manually digitized archives

These documents may require preprocessing during ingestion.

---

# 4. Master Ledger

The master ledger records all documents processed by the pipeline.

File location:

```
data/manifests/jae_master_ledger.csv
```

Each row represents one document.

---

## Ledger Fields

| Field            | Type    | Description                       |
| ---------------- | ------- | --------------------------------- |
| doc_id           | string  | unique document identifier        |
| title            | string  | article title                     |
| authors          | string  | author list                       |
| journal          | string  | publication source                |
| year             | integer | publication year                  |
| volume           | string  | journal volume                    |
| issue            | string  | journal issue                     |
| pages            | string  | page range                        |
| route            | string  | ingestion route (modern / legacy) |
| filepath         | string  | relative path to raw file         |
| ingest_status    | string  | ingestion completion status       |
| embedding_status | string  | vectorization completion status   |
| analysis_status  | string  | analysis completion status        |

---

# 5. Processed Text Outputs

Cleaned text outputs are stored in:

```
data/processed/
```

Each processed document generates:

```
doc_id_clean.txt
```

These files represent normalized text used for embedding.

---

# 6. Embedding Outputs

Vector embeddings are stored as serialized arrays.

Expected format:

```
doc_id_embedding.npy
```

Storage location:

```
data/processed/embeddings/
```

Each file contains a vector representation of the document generated by the embedding model.

---

# 7. Embedding Metadata

Associated metadata may include:

| Field          | Description               |
| -------------- | ------------------------- |
| doc_id         | document identifier       |
| token_count    | tokens processed          |
| model          | embedding model used      |
| embedding_dim  | vector dimensionality     |
| compute_device | MPS / CPU                 |
| timestamp      | embedding generation time |

---

# 8. Analysis Outputs

Analytical outputs are stored in:

```
data/processed/analysis/
```

Expected artifacts include:

* centroid vectors
* temporal drift measures
* clustering assignments
* semantic distance matrices

Example files:

```
epoch_centroids.csv
semantic_drift_vectors.npy
cluster_assignments.csv
```

---

# 9. Temporal Segmentation

Documents are grouped into temporal epochs using:

```
EPOCH_STEP = 5
```

Range:

```
1960 – 2026
```

Epoch example:

```
1960–1964
1965–1969
1970–1974
```

This enables longitudinal vector-space analysis.

---

# 10. Data Integrity Rules

The pipeline enforces several integrity constraints.

Rules include:

* each `doc_id` must be unique
* all raw documents must appear in the master ledger
* embedding outputs must correspond to ledger entries
* analysis outputs must reference existing embeddings

Pipeline phases will halt if violations are detected.

---

# 11. Naming Conventions

Document identifier format:

```
JAE_YEAR_VOLUME_ISSUE_PAGE
```

Example:

```
JAE_1998_39_2_45
```

This ensures deterministic document identification across pipeline phases.

## Structured Section Export Schema

Structured JSON outputs contain flattened section fields.

| Field | Type | Description |
|------|------|-------------|
| doc_id | string | Stable document identifier |
| route | string | Route_A_Modern or Route_B_Legacy |
| year | int | Publication year |
| extraction_method | string | fitz or ocr |
| page_count | int | Page count |
| raw_text_length | int | Length before cleaning |
| clean_text_length | int | Length after cleaning |
| segmentation_strategy | string | Segmenter strategy used |
| A_intro | string | Introduction section |
| A_methods | string | Methods section |
| A_results | string | Results section |

### Section Key Constants

The canonical section identifiers are defined in:

bins/s04_utils/schemas.py
A_INTRO = "A_intro"
A_METHODS = "A_methods"
A_RESULTS = "A_results"
